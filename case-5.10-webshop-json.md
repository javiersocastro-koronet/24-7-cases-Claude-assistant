# Case 5.10 — Webshop JSON
**Pattern:** C — Data Correction | **Guidebook section:** 5.10

---

## ⚠️ Pattern C warning — read before proceeding

1. Default to the ABS interface, not the database
2. Get verbal confirmation from the customer before any destructive action
3. If the agent has never run this procedure before, ask Tier 2 or Martijn to walk through it first

---

## Trigger phrases — what the customer says

| Phrase | Time of call | Probability |
|---|---|---|
| "webshop is empty" / "lots disappeared from webshop" | Any | Very high |
| "webshop shows nothing" / "buyers can't see any lots" | Any | Very high |
| "JSON error on webshop" / "webshop broken after lot update" | Any | High |

---

## Step-by-step resolution

Follow in order. This procedure corrects a malformed lot record that is causing a JSON parsing failure on the webshop. Make all corrections through the ABS interface — not directly in the database.

**Step 1 — Reproduce the empty webshop and locate the error**
Ask the customer (or agent, if they have webshop access) to:
1. Open the webshop and reproduce the empty/broken selection
2. Click the **error link** that appears on the webshop page — this opens a raw JSON response

**Step 2 — Copy the raw JSON**
Copy the full JSON from the error response.

**Step 3 — Decode the JSON**

> ⚠️ **PENDING — confirm before using in a live call:**
> Confirm the exact URL of the JSON decoder tool used by the team. The guidebook references a decoder that converts Unicode escapes (e.g., `-` → `-`) but does not specify the URL.
> *Confirm with: Martijn | Added: 2026-06-08*

Paste the JSON into the decoder tool. The decoder converts Unicode escape sequences (e.g., `-` → `-`) into readable characters. Copy the decoded output.

**Step 4 — Validate the decoded JSON to find the error location**

> ⚠️ **PENDING — confirm before using in a live call:**
> Confirm the exact URL of the JSON validator tool used by the team.
> *Confirm with: Martijn | Added: 2026-06-08*

Paste the decoded JSON into the validator. The validator identifies the **line number and character position** of the parsing error. Note this location.

**Step 5 — Identify the problematic field near the error position**
In the decoded JSON, navigate to the line/position reported by the validator.
Identify the **key field** (lot identifier field) near the error. This field corresponds to a lot record in ABS that contains an illegal character (typically a dash `-` or another character that breaks JSON parsing).

**Step 6 — Look up the internal lot number in the database**

> ⚠️ **PENDING — confirm before using in a live call:**
> Confirm the exact lot lookup query — specifically the table name, field names, and the correct join or filter to go from the webshop key field to the ABS lot number.
> *Confirm with: Martijn | Added: 2026-06-08*

Run a query on the ABS database to get the internal lot number corresponding to the key field identified in Step 5.

**Step 7 — Open the lot in ABS and fix the character**
Open the lot in the **ABS interface** using the internal lot number.
Locate the field containing the illegal character (e.g., a dash in a field that does not allow it).
Remove the dash or fix the character. Save the record.

> ⚠️ Always fix in the ABS interface, not directly in the database. ABS save triggers perform additional processing (indexing, cache invalidation) that a direct DB update bypasses.

**Step 8 — Refresh the webshop and confirm**
Ask the customer to refresh the webshop. Confirm that lots are now visible.

**Step 9 — Cart variant (if the issue is on a cart/order row, not a lot)**
If the broken record is an order row rather than a lot:
- Use the **order row key** instead of the lot key to locate the record
- The fix procedure is the same — open in ABS interface, correct the character, save

> ⚠️ **PENDING — confirm before using in a live call:**
> Confirm the exact procedure for the cart variant — specifically how to navigate to an order row by key in the ABS interface.
> *Confirm with: Martijn | Added: 2026-06-08*

**Step 10 — Escalate to Tier 2**
If the error location cannot be identified, or if fixing the character does not resolve the empty webshop:
Tell the agent: "Proceed to escalate to Tier 2."
Brief: customer name, JSON error line/position, field identified, what was changed in ABS, result after save.

---

## Important nuances

> ⚠️ The webshop renders its catalogue from a JSON feed generated by ABS. A single malformed character in any lot record can cause the **entire JSON to fail to parse**, making the entire webshop appear empty. The fix is always to find and correct that one character.

> ⚠️ Always fix in the **ABS interface**, not directly in the database. Direct DB edits bypass ABS save triggers, which handle indexing and cache invalidation. A direct edit may fix the JSON but leave the lot in an inconsistent state.

> ⚠️ The most common offending character is a **dash (`-`)** in a field that does not allow it. This often gets introduced when lot data is copy-pasted from external sources.
